## 1) Software engineering fundamentals

- **Software engineering** applies scientific principles to the design and creation of software.

- Software engineers are responsible for **designing, building, and maintaining** software systems.

## 2) Why the SDLC matters

Using the **Software Development Life Cycle (SDLC)** can improve efficiency and reduce risk by:

- Clarifying **who should work on what, and when**

- Improving communication between **customers, stakeholders, and the development team**

- Helping stakeholders understand **where they fit in the process**

- Supporting handoffs so cross-domain teams can move work to the **next phase**

## 4) Requirements and key documents

### Requirements gathering includes:

- Identifying stakeholders

- Establishing goals and objectives

- Eliciting requirements

- Documenting, analyzing, prioritizing, and confirming requirements

### Common documents:

- **SRS:** Captures system functionalities and benchmarks/service levels for performance.

- **URS:** Subset of the SRS that details user specification requirements.

- **SysRS:** Contains the same information as an SRS and may also include capabilities, interfaces, user characteristics, policy/regulatory/personnel requirements, performance and security requirements, and acceptance criteria.

## 5) SDLC methodologies

- **Waterfall, V-shape**, and **Agile** are different approaches to implementing the SDLC.

## 6) Testing overview

- **Functional testing:** Verifies inputs and expected outputs.

- **Non-functional testing:** Verifies attributes such as performance, security, scalability, and availability.

- **Regression testing:** Confirms changes (e.g., bug fixes) do not break existing functionality.
